2. Сохранение в структуру и возможность выбора пользователем
	2.1. По поводу сохранения можно сказать что мне нужно сохранять только два параметра это путь и объем. С путями я уже разобрался, а вот размер USB накопителя пока что остается вопросом.
	2.2. Итак. В Linux, чтобы узнать размер USB накопителя необходимо выполнить команду
	cat /sys/block/sdX/size
	где X - это конкретная деректория USB устройства.
	В данном файле написано общее количество секторов на данном диске
	В Linux и архитектуре хранения данных сектор является минимальной единицей адресации на носителе.
	Для ядра Linux в /sys/block размер одного логического сектора всегда жестко принят за 512 байт.
	Именно поэтому для размера был выбран тип данных unsigned long long 
	К примеру на моей тестовой флешке количество секторов равно 15671296 
	Следовательно умножив 15671296 на 512 получаем 8023703552 Байт
	Это в свою очередь равно 8023703552 байт = +- 7,47 Gb 	(Следовательно продавцы флешек обманывают покупателей :))
	в двоичной системе
	1 ГБ = 2 в 30 степени = 1073741824 байт

3. Очистка USB устройства
	Очистку будем выполнять пока что в один проход. А именно перезаписью всех данных на нули.
	Данный метод уничтожит файловую систему USB накопителя. Что сделает восстановление данных труднее в разы.
	Необходим системный вызов write().
	write(файловый дескриптор, указатель на буфер памяти содержащий данные для записи, количество байт которые требуется записать).
	Так же как и любая другая функция write() возвращает значение успеха и неудачи (например 1 и -1)
	Поэтому мне нужна переменная для поиска ошибки (если она случится). Для такой переменной я использую тип ssize_t - signed size_t т.е (знаковый) т.е может быть отрицательным 
	Для начала необходимо дать путь к нашему USB устройству (path в структуре не тот что нужен поскольку /sys/block обеспечивает информацией о накопителях и не служит для записи данных)
	Нужен каталог /dev/sdX
	где X - это собственно USB устройство
	Поэтому нужно вернутся к пункту 2 и склеить новый путь path. Получается ...,"/dev/%s", entry -> d_name
	Готово путь есть.
	Поскольку затираем нулями то так же необходим memset() - заполнения блока памяти определенным значением. <string.h>
	memset(указатель на начало области памяти которую нужно заполнить, значение для заполнения каждого байта, количество байт к заполнению)
	для начала открываем файл согласно пути из структуры
	<fcntl.h> - это заголовочный файл стандартной библиотеки C, предоставляющий функции и макросы для управления файловыми дескрипторами.
	int fd (file descriptor) - возвращенное значение функцией open(). значение по умолчанию 1
	int fd = open (usb->path, O_WRONLY);
	O_WRONLY - флаг который указывает что файл открыт исключительно для записи
	Итак теперь необходимо подготовить буфер для записи и заполнить его нулями
	<unistd.h> - заголовочный файл предоставляющий доступ к прикладному программному интерфейсу POSIX 
	Есть два варианта как это сделать 
	через стек и через кучу 
	через стек сокращается время поскольку процессору не требуется искать свободное место для буфера
	Однако есть минус. Размер стека ограничен, следовательно мы не сможем постоянно увеличивать размер буфера например для увеличения скорости записи на USB 3.0
	Через кучу память так же ограничена однако она ограничена оперативной памятью компьютера а это примерно 16 - 32 ГБ в среднем т.е это делает возможным увеличивать буфер а с ним и скорость записи на быстрые USB устройства. Из минусов то что требуется время на выделение памяти под сам буфер.
	Далее пишем цикл который будет собственно "cтирать данные" с USB устройства.
	Необходимо:
		подстчет записанных байтов
		проверка на доступное место на носителе
		проверка на оставшееся количество неперезаписанных байтов
		fsync(fd) - принудительно записывает все измененные данные и метаданные файла из кэша операционной системы на физический носитель (т.е запись на чипы носителя) следовательно это является долгой чисткой с очень затрудненной возможностью к восстановлению
